Ready Snippet

可直接放入 workflow 的修复片段

下面的配置使用 step output 输出小写 owner，既能正常运行，也能减少 GitHub 编辑器对 env 动态变量的误报。

!
为什么会失败？

Docker 镜像仓库名必须全部小写，johnWu3128 中的大写 W 会导致 buildx 拒绝构建。

✓
推荐怎么修？

把 GITHUB_REPOSITORY_OWNER 在 bash 中转成小写，再拼接到 ghcr.io 镜像地址里。

build-docker-image.yml 
- name: Set lowercase image owner
  id: image
  run: echo "owner=${GITHUB_REPOSITORY_OWNER,,}" >> $GITHUB_OUTPUT

- name: Build and push Docker image
  uses: docker/build-push-action@v7
  with:
    context: .
    push: true
    platforms: linux/amd64,linux/arm64
    tags: |
      ghcr.io/${{ steps.image.outputs.owner }}/deploy20260601:latest
    labels: |
      org.opencontainers.image.source=https://github.com/${{ github.repository }}
      org.opencontainers.image.description=HTTP Server
      org.opencontainers.image.licenses=MIT
    cache-from: type=gha
    cache-to: type=gha,mode=max

FAQ

常见问题

围绕 GHCR、Docker tag 和 GitHub Actions 的几个高频问题。

GitHub 用户名可以有大写，为什么镜像不行？

GitHub 显示名允许大写，但 Docker distribution 规范要求 repository name 使用小写字符，所以 GHCR 镜像路径也需要小写。

可以直接写死 johnwu3128 吗？

可以。如果仓库只属于你个人账号，直接写死小写 owner 是最简单的方案；如果要复用到组织或其他账号，建议用脚本自动转换。

pull_request 时一定要 push 镜像吗？

通常不建议。生产项目可以把 push 改成仅 main 分支推送，PR 只构建不发布，避免无意义镜像污染仓库。

发布后在哪里看镜像？

进入 GitHub 仓库或个人主页的 Packages 页面，可以看到 ghcr.io 下发布的容器镜像包。

让下一次发布少一点猜测，多一点确定。

复制修复片段、提交 workflow、重新运行 Actions。只要镜像路径保持小写，buildx 就能顺利进入真正的构建阶段。
